## 1. 物理隔离：git worktree 方案

主 worktree（main 分支）保持 `~/Coding/Archer/Chameleon/` 原位置不动；4 个 agent worktree 放 `.worktrees/`（已 .gitignore）：

```
~/Coding/Archer/Chameleon/                ← 主 worktree (main) - 协调 / merge train
├── .worktrees/                            ← 4 个 agent 独立工作区
│   ├── workflow-engine/                    Agent A → feat/v1.1-workflow-engine
│   ├── rag-deepen/                         Agent B → feat/v1.1-rag-deepen
│   ├── billing-gateway/                    Agent C → feat/v1.1-billing-gateway
│   └── ui-deepen/                          Agent D → feat/v1.1-ui-deepen
└── ... (其它项目文件)
```

**优点**：
- 每个 worktree 是独立工作目录，独立 .venv / node_modules / dev server / git index
- 共享同一 `.git/`（merge / rebase / 分支切换全部正常）
- 一次 `git worktree remove` 干净清除
- 每个 agent 看到的 `git status` 只含自己的改动

## 4. 跨 Agent 协议

### 4.1 通信渠道

- **每日同步**：每个 agent 完成一个 PR → push branch → 主 worktree（main）做 merge train
- **schema 改动公告**：Agent C 改 ORM 时，在 commit message 顶部加 `[SCHEMA-CHANGE]` 标记，ABD 看到后立刻 rebase
- **API 接口对齐**：Agent C/A/B 暴露新 API 时**先在 `docs/spec/v1.1-apis.md` 写 OpenAPI 片段**，Agent D 看着 spec 写 mock + 真接口切换

### 4.2 Merge Train（每天主 worktree 跑）

```bash
# 主 worktree 主线
cd ~/Coding/Archer/Chameleon
git fetch --all
# 顺序：C（schema）→ A → B → D（前端最后切真接口）

# 解冲突时优先保留 main 已有逻辑，agent 改动再 review
```

```bash
cd .worktrees/<my-agent>

# 解冲突，继续干
```

## 7. 资源管理（防 4 个 agent 跑爆机器）

- **同时跑后端 dev server 数量**：建议 ≤ 2 个（每个 backend ≈ 800MB；4 个 ≈ 3.2GB）→ 各 agent 只在 e2e 验证时启 dev，平时 pytest 即可
- **同时跑 frontend dev server 数量**：建议 ≤ 2 个（Vite HMR ≈ 500MB 内存）
- **PostgreSQL**：单实例 4 个 DB OK（共享 connection pool）
- **Redis**：单实例多 prefix（已隔离）
- **Chrome MCP**：每 agent 自己起 tab group（用 `tabs_create_mcp`），不抢主 group

如果 CPU / 内存吃紧：D（前端）+ B（RAG，无 dev server 强依赖）可以共享一台机器，A / C（重 dev server）各自独占。

---

**核心信念**：4 个 agent 并行 = 同样 8 周完成 v1.1 时打的深度 ~3x 单线程；前提是文件域严格隔离 + Agent C 的 schema 改动同步及时。

## 3. 借鉴方案细化（Tier 1 六项）

### T1-1 · GraphEngine Worker Pool + Ready Queue

**为什么**：Dify GraphEngine 是 4.8K 行的模块化并发引擎；我们 320 行串行 BFS，根本跑不动 100+ 节点的真实 workflow，更别说 Parallel / Iteration。

**借鉴路径**（不直接 cp，重写）：
- `/Users/links/Coding/Hub/dify/api/core/workflow/graph_engine/` 整个目录架构（每个子目录一个职责）
- 切到 Python asyncio：`asyncio.Queue` 替代 Dify 的 `queue.Queue`；`asyncio.gather + Semaphore` 实现 Worker Pool

**Chameleon 化设计**（符合 MVC + Result 包装）：
```
core/graph/engine/
  ready_queue.py      入度归零的节点入队
  worker_pool.py      asyncio.Semaphore + create_task 并发
  event_manager.py    SSE 事件分发到 SSEEventKind registry
  orchestrator.py     主循环：拉队列 → 调度 worker → 收事件 → 推前端
  state.py            执行状态机 + 变量池（VariablePool）
```

**红线**：保持现 `Node` 抽象 API 兼容；旧的 `executor.py` 标 deprecated 一个 minor 版本后删。

### T1-2 · Pre-consume + Post-consume 计费

**为什么**：v1.0 声称多租户 + 配额，但用户可以"花到负数"——没有预扣机制。SaaS 必死。

**借鉴路径**（重写）：
- `/Users/links/Coding/Hub/one-api/relay/controller/helper.go:68-150`：preConsumeQuota = base(500) + promptTokens + maxTokens，按 ratio 乘 → Redis decr → 实际 token 出后 → 差额返还

**Chameleon 化设计**：
```
core/observe/billing.py
  estimate_cost(request) -> estimated_usd     # base + prompt + max_tokens × ratio
  pre_consume(workspace_id, estimated_usd)    # Redis INCRBY quota_used / atomic
  post_consume(workspace_id, actual_usd)      # 实际 - 预扣 = 差额，> 0 退还、< 0 补扣
  trust_threshold(workspace_id) -> bool       # quota_remaining > 100 × estimated 时跳过预扣
```

**红线**：
- 一定走 Redis 原子操作（Lua script 或 WATCH/MULTI），不要 SQL ROW LOCK
- 失败要 fallback：Redis 不可达时降级到 SQL ROW LOCK
- record_call 末尾 `await billing.post_consume(...)`

### T1-3 · LLM 节点流式 + Memory buffer

**为什么**：v1.0 的 LLMNode 一次性等完整 response，体感卡死；没有 memory buffer，多轮上下文每次重传。

**借鉴**：Dify `api/core/workflow/nodes/llm/node.py` (1414 行)

**Chameleon 化**：
```
core/graph/nodes/llm.py
  - stream=True 时走 provider.stream → 边出边 yield NodeStreamEvent(chunk=...)
  - memory_buffer (Optional[str])：从 messages 表按 conversation_id + window=10 拉历史
  - tool_calls 多轮：parse function_call → 调 ToolNode → 把结果塞回 messages 再调 LLM
```

### T1-4 · Iteration / Parallel / HumanInLoop 节点

**Iteration 节点**（借鉴 Dify `list_operator`）：
```
class IterationNode(Node):
    items: list                    # 来源变量
    max_iterations: int = 100      # 防爆栈
    parallel_limit: int = 5        # 并发上限
    early_stop_expression: str    # Jinja2 表达式
    body_subgraph: SubgraphSpec    # 嵌套子图
```

**Parallel 节点**（fork-join）：
```
class ParallelNode(Node):
    branches: list[SubgraphSpec]  # N 个分支
    join_strategy: Literal["all", "any", "first"]
```

**HumanInLoop 节点**（持久化等待）：
```
class HumanInLoopNode(Node):
    timeout_seconds: int          # 超时降级
    notify_channels: list         # Slack / 邮件
    # 实现：写一行到 human_input_pending 表 → 暂停 graph → 用户回填后 / 超时后 → 恢复
```

### T1-5 · Group 倍率系统

**借鉴**：One-API `relay/billing/ratio/group.go` GroupRatio map

**Chameleon 化**：
- 新表 `workspace_groups(workspace_id, group_code, ratio_multiplier)`：standard=1.0 / premium=0.8 / trial=2.0
- `system/pricing/service.py:calc_cost(...)` 改签名加 `group_ratio: Decimal`
- 路由层从 `current_workspace` 解 group → 查倍率 → 算 cost

### T1-6 · Multi-query expansion / HyDE 召回

**为什么**：HybridPipeline 6 步看起来全了，但首步 vector 召回是单 query；用户 query 短或歧义时召回率塌方。FastGPT 标准做法是 LLM 改写 + 多 query 并行 + 结果融合。

**借鉴**：FastGPT `packages/service/core/dataset/search/` query rewrite + HyDE

**Chameleon 化**：
```
core/retrieval/expander.py
  rewrite_query(q, llm_client) -> list[str]   # LLM 输出 3-5 个 query 变体
  hyde_query(q, llm_client) -> str            # LLM 生成"假答案"，反向 embed

HybridConfig
  + multi_query_count: int = 0    # 0 表示关闭
  + use_hyde: bool = False
```

把所有变体 query 跑一遍 vector + RRF 融合，单 query 召回率提升 15-30%（FastGPT 实测数据）。

## 5. 借鉴时的红线（再申明一次）

每条增强项动手前过 [[coding-standards]] § ⑤ 三问筛子：

1. **符合 MVC 吗**？
   - Dify 把业务塞 controller / API；One-API 把计费塞 middleware → 都要拆到 Chameleon 的 service 层
   - GraphEngine 重写时不要把节点执行写进 router
2. **能用 Python 优雅实现吗**？
   - Dify TS-flavor 的 EventEmitter 改成 Python asyncio + dataclass event
   - One-API Go-flavor 的 in-memory map 改成 Redis + 异步刷新
3. **会引入新依赖吗**？
   - Tier 1-3 整个清单**只有以下新依赖**（写 ADR）：
     - `bge-reranker` HF 客户端 / `cohere` SDK（T2-6）
     - `tiktoken`（T2-7 token-aware chunker）
     - `react-window`（T3-4 虚拟列表，前端）
   - **不引入** ClickHouse / Kafka / Milvus / Firecracker — 这些是 LangFuse / RagFlow 用的，对 Chameleon "单 PG 路线" 是反模式

**冲突时永远以 Chameleon 规约优先**：
- API 零业务 / Service 不返 ORM / Result.ok 包装 / GET+POST / 严格 MVC
- 不直接 cp 源码，借**概念 / 表结构 / 算法**重写

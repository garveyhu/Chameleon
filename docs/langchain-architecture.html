LangChain 架构与抽象模式深度解析

从「一切皆 Runnable」到回调总线——拆解 LangChain 如何用一组抽象基类，把异构的 LLM 生态收口成可组合、可观测、可替换的统一形状。

学习笔记 · 2026-05-29

LangChain 真正的价值不是「调 OpenAI 的语法糖」，而是一套架构契约：它赌定了三件事—— 统一接口、可组合、可观测内建，并用一组抽象基类把这三件事钉死在最底层。 理解了这套抽象，你就理解了为什么「在 LangSmith 配一下，全链路流量就被捕获并渲染」。

它要解决的根本矛盾是：LLM 生态极度异构——不同厂商的模型、不同的向量库、五花八门的检索器/工具/解析器， 还要支持同步/异步/流式/批量四种执行形态，外加重试、回退、缓存、追踪这些横切关注点。 如果每个组件各长各的样、每个调用方各写各的胶水，系统会迅速腐烂。LangChain 的答案是：先定义形状，再填实现。

一句话主线：把每一种「能力」抽象成一个基类，让基类承担所有横切逻辑（回调、配置、重试、流式）， 子类只填一个「核心方法」；再让所有基类共享同一个执行协议（Runnable）和同一条回调总线（CallbackManager）。 于是组合是免费的，观测是免费的，替换实现也是免费的。

1第一性原理：一切皆 Runnable

LangChain 的地基是一个协议——Runnable。模型、提示词模板、检索器、输出解析器、工具、甚至一段普通函数， 统统实现同一个接口。这意味着任意两个组件都能拼接，因为它们说同一种「语言」。

这套统一接口的核心方法（每个都有同步 / 异步两版）：

方法形态说明
invoke / ainvoke单次一个输入 → 一个输出
batch / abatch批量多输入并行处理
stream / astream流式逐块吐出输出
astream_events事件流吐出结构化的执行事件（含 run_id / 嵌套层级）
with_config / with_retry / with_fallbacks / bind装饰包一层返回新的 Runnable（不可变）
关键直觉：统一接口不是为了「好看」，而是组合的前提。 当所有东西都是 Runnable，「编排」就退化成「拼接」——不需要专门的胶水类，| 运算符就够了。

2LCEL：用 | 组合一切

LCEL（LangChain Expression Language）就是建立在 Runnable 之上的声明式组合语言。 管道符 | 把左边的输出喂给右边，串成一个 RunnableSequence；而这个序列本身又是一个 Runnable （组合子的封闭性），可以继续被组合、被流式、被追踪。

# 最经典的三段式：提示词 → 模型 → 解析器
chain = prompt | model | output_parser
chain.invoke({"topic": "变色龙"})        # 同步
async for chunk in chain.astream({...}):  # 流式——整条链自动支持
    ...

除了串行，还有几个组合原语（它们也都是 Runnable）：

原语作用
RunnableSequence（a | b）串行：上一步输出 = 下一步输入
RunnableParallel（{}）并行：同一输入分发到多个分支，输出合成 dict
RunnablePassthrough原样透传输入（常配 .assign() 往字典里加字段）
RunnableLambda把任意函数包成 Runnable
RunnableBranch条件路由（if/elif/else）

一条 RAG 链用 LCEL 写出来，结构一目了然：

rag = (
    {"context": retriever, "question": RunnablePassthrough()}  # 并行：检索 + 透传问题
    | prompt        # 把 context+question 填进模板
    | model         # 调 LLM
    | StrOutputParser()
)
组合子封闭性（closure）：「Runnable 组合出来的东西仍是 Runnable」——这是整套设计的数学美感所在。 正因如此，流式、批量、回调、配置传播这些能力，对一条链和对一个组件是一视同仁的，不需要为「链」单独写一套。

3抽象基类全景

在 Runnable 这个通用协议之下，LangChain 为每一类「能力」定义了专门的抽象基类。 基类规定了「这一类能力的标准形状」+「你必须实现的那个核心方法」。下面是主干：

每个基类的「契约」长这样——你只填核心方法，公共方法基类替你实现：

能力抽象基类你实现的核心方法你调用的公共方法
对话模型BaseChatModel_generate / _agenerateinvoke / stream
检索器BaseRetriever_get_relevant_documentsinvoke
工具BaseTool_run / _aruninvoke / run
输出解析BaseOutputParserparseinvoke
提示词BasePromptTemplateformatinvoke
向量库VectorStoresimilarity_search 等as_retriever() → Retriever
嵌入Embeddingsembed_documents / embed_query（直接调，无回调）
回调BaseCallbackHandleron_*_start/end/error（被总线调用）
注意这个不对称：Embeddings 是少数不在回调体系里的基类—— 它不 fire 生命周期事件。所以「向量化」天生是 LangChain 追踪的盲区，要观测它得自己包一层。 （这正是 Chameleon 方案 A 里 Embedding/Reranker 走自有 observe() 切面、而非 LangChain 回调的原因。）

4模板方法：脏活在基类，子类只填核心

这是整套设计里复用性最高的模式。以 BaseChatModel 为例：你调的是公共的 invoke()， 但 invoke() 自己并不知道怎么调某个具体模型——它把横切逻辑全做了（解析配置、开回调、缓存、计时、错误处理）， 然后在中间留一个抽象方法 _generate() 让子类填厂商细节。

# 你要接一个新模型，只需继承基类、实现一个方法
class MyChatModel(BaseChatModel):
    def _generate(self, messages, stop, run_manager, **kwargs):
        # ← 只写「怎么调我家模型」这一件事
        resp = my_vendor_sdk.chat(to_vendor_format(messages))
        # run_manager 是基类塞进来的「回调把手」，用它发 token 事件
        return ChatResult(generations=[...])
    # invoke / stream / batch / 回调 / 重试 …… 统统白嫖基类的
为什么这招威力大：因为回调被埋在基类的公共方法里，所以「任何子类、任何调用方」都自动 fire 事件—— 观测不依赖子类作者记得埋点，也不依赖调用方配合。这就是「在底层切面做好，上层全白嫖」的本质。 你只要走的是 BaseChatModel，追踪就是免费的。

5回调系统：可观测性内建于基类

这是和 Chameleon 当前工作最相关的一节，也是 LangSmith 魔法的来源。回调系统由三部分组成：

  • HandlerBaseCallbackHandler——定义一堆生命周期钩子，你想消费事件就实现它。
  • ManagerCallbackManager——总线。持有一组已注册的 handler，在每个生命周期点把事件 fan-out 给所有 handler。
  • Run每次组件执行 = 一个 run，分配 run_id，并记下 parent_run_id，从而拼出一棵运行树。

BaseCallbackHandler 的钩子覆盖了所有组件类型——这就是「全捕获」的接口基础：

class BaseCallbackHandler:
    def on_chat_model_start(self, serialized, messages, *, run_id, parent_run_id, **kw): ...
    def on_llm_new_token(self, token, *, run_id, **kw): ...   # 流式逐 token
    def on_llm_end(self, response, *, run_id, **kw): ...      # 含 token 用量
    def on_chain_start(self, ...): ...  def on_chain_end(self, ...): ...
    def on_tool_start(self, ...): ...   def on_tool_end(self, ...): ...
    def on_retriever_start(self, query, ...): ...
    def on_retriever_end(self, documents, ...): ...
    def on_agent_action(self, ...): ...  def on_agent_finish(self, ...): ...

5.1 run 树：parent_run_id 怎么拼出层级

每个组件被 invoke 时，基类向 CallbackManager 要一个 run manager（带新分配的 run_id）。 当这个组件内部又调别的 Runnable 时，子调用从配置里继承父的回调管理器，于是子 run 的 parent_run_id = 父 run 的 run_id。 一棵嵌套的运行树就这样自动长出来——没有人手动维护层级。

回调如何「传播」下去？靠 RunnableConfig。你在最外层 invoke 时塞一个 handler，它会随 config 流到每一个子 Runnable：

handler = MyHandler()
rag.invoke(question, config={"callbacks": [handler]})
# handler 会收到 on_chain_start → on_retriever_* → on_chat_model_* → on_tool_* …… 一条不漏

5.2 LangSmith 不过是「一个 handler」

理解了上面，LangSmith 的「配一下全捕获」就毫无神秘可言了：LangSmith 的 tracer（LangChainTracer）本身就是一个 BaseCallbackHandler。它实现了所有 on_* 钩子，把每个 run（带 run_id/parent_run_id/输入输出/用量/计时）POST 到 LangSmith 服务端， 服务端再把运行树渲染成那个漂亮的瀑布图。

而「配一下」= 设一个环境变量 LANGCHAIN_TRACING_V2=true，LangChain 就在全局回调管理器里自动挂上这个 tracer。 于是你一行业务代码都不用改，所有走 Runnable 的流量就被捕获了。

对 Chameleon 的映射：你们的 GenerationRecorder 就是「自己写的那个 handler」——它实现 on_llm_end 把 generation 落进 call_logs。 方案 A 要做的，无非是让 Retriever / Tool 也走这条总线（成为 LangChain 原生基类）， 再把 GenerationRecorder 泛化成处理 on_retriever_end / on_tool_end 的多事件 handler。 顺带——挂上 LangSmith 官方 tracer 就能云端渲染，二者并存。

6依赖倒置与分包：核心定义形状，集成填实现

LangChain 把「抽象」和「实现」拆进不同的包，方向严格单向——这是它能容纳上百个厂商集成却不烂的关键。

包职责
langchain-coreRunnable、所有抽象基类、回调协议、消息/文档等数据结构。只定义形状，不绑厂商。
langchain-openai 等伙伴包把某厂商 SDK 适配成基类的实现（Adapter）。
langchainchains / agents / 检索策略等编排逻辑，只依赖 core 的接口。
langchain-community长尾社区集成。
分层心法：把「契约」沉到一个零依赖的核心包，让所有实现反向依赖它。 换厂商 = 换一个实现包，编排逻辑一行不动。这与 Chameleon 的 chameleon-core ← system ← api 单向依赖是同一种纪律—— 也正因如此，方案 A 里「核心层只能定义基类 + sink 协议，不能 import record_call」。

7流式与事件：一次实现，处处可流

因为流式被定义在 Runnable 协议里（astream），而组合子是封闭的，所以一条链自动可流式—— LangChain 会尽可能把流「透传」到最后一个支持流的环节。更强的是 astream_events：它吐出结构化事件 （on_chat_model_stream / on_retriever_end …，每个都带 run_id、parent_ids、tags）， 让前端可以按节点粒度渲染「正在检索…正在生成…」。

async for ev in rag.astream_events(question, version="v2"):
    if ev["event"] == "on_retriever_end":
        show_citations(ev["data"]["output"])
    elif ev["event"] == "on_chat_model_stream":
        push_token(ev["data"]["chunk"])

注意：astream_events 本质上也是回调系统的产物——它在内部挂一个把事件转成异步迭代器的 handler。 所以「流式事件」和「追踪」共享同一套底层机制。

8设计模式总览：把 LangChain 翻译成 GoF

抛开框架术语，LangChain 的抽象其实是几个经典设计模式的精到组合：

LangChain 机制经典模式解决什么
基类 invoke() 夹 _generate()模板方法 Template Method横切逻辑只写一次，子类只填核心
集成包实现基类接口策略 / 适配器 Strategy / Adapter厂商可插拔、可替换
RunnableSequence/Parallel组合 Composite「链」和「组件」被一致对待
BaseCallbackHandler + Manager观察者 Observer执行与观测解耦，多消费端并存
with_retry / with_fallbacks / bind装饰器 Decorator不改组件给它加能力（返回新 Runnable）
langchain-core 定接口依赖倒置 DIP实现依赖抽象，而非反过来

9提炼：可借鉴的架构心法

抛开 LangChain 具体 API，留下这几条放之四海皆准的设计心法：

  1. 先定形状，再填实现。给每类能力一个抽象基类 + 一个统一执行协议，异构生态才能被「拍平」成可组合的积木。
  2. 横切关注点钉在基类，别散在调用方。回调、配置、重试、流式——谁都不该手写。基类做一次，所有子类与调用方白嫖。（这正是你说的「写入 call_log 要在更底层做好」。）
  3. 可观测性是一等公民，不是事后补丁。把「发事件」织进基类的生命周期，再让消费端（落库 / 云端 / 前端流）作为可插拔 handler 挂在同一条总线上。
  4. 组合优于继承，且组合子要封闭。「组合出来的还是同类」才能让能力（流式/追踪）对组件和对链一视同仁。
  5. 契约沉核心，实现反向依赖。零依赖的核心包定义接口，集成与应用都指向它——换实现零成本，分层不腐烂。
回到 Chameleon：你们已经在 LLM 这条做对了全部五条（BaseLLM = 基类、GenerationRecorder = handler、core 定契约）。 方案 A 不过是把同样的五条心法，从 LLM 一种能力复刻到 Retriever / Tool / Embedding / Reranker—— 让「任意入口、任意组件，自动产出干净 trace」从 LLM 的专属待遇，变成全组件的默认行为。
